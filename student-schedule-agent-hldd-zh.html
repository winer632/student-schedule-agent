1. 产品愿景

概述

这个产品是一个面向学生的 AI 日程智能体。它把学校作业、成绩、固定安排、个人偏好以及临时变动整合进一个主日程。它不只是展示任务列表，而是主动推理工作应该安排在什么时候，解释原因，在不确定时提问，并根据用户调整逐渐学习。

第一版专注于学校场景，尤其是与 Schoology 相关的工作流。长期来看，它可以扩展成任何人都能使用的任务与时间规划助手（迁移路径见第 16 节）。

一句话定位

MVP：把手动/粘贴录入的作业与固定安排，通过确定性排程 + LLM 拆分转化为动态周日程，并能通过聊天修改计划、解释原因、可撤销。

愿景（路线图）：成绩感知优先级计算、随使用学习偏好、以及从日历自动聚合——这些差异化能力在 MVP 中为只读示例或未落地，不计入验收（见第 9 节、第 11 节、第 16 节）。（真实 Schoology 作业/截止日期导入已属 MVP，见第 7 节）

目标用户

主要 persona

  • 使用 MacBook 完成学习管理的高中生，通常会从 Schoology、教学大纲、PDF 或老师公告中查看作业。
  • 同时有 AP/荣誉课程、体育训练、社团或家庭安排，希望在电脑上集中规划本周任务。
  • 已有基础日历或待办使用习惯，但需要帮助把作业拆解、安排到可用时间，并在计划被打乱后重排。
边界

非目标 persona 与数据限制

  • MVP 是单用户 Mac 桌面应用，不服务家长端、老师端、学校管理员端，也不做班级级数据分析或多人协作。
  • 不做云账号、不做多设备同步、不做 Web/SaaS 管理后台。
  • 数据默认保存在用户本机；MVP 接入真实 Schoology（官方 API 优先、浏览器插件兜底）导入课程/作业/截止日期（成绩只读展示）；外部日历同步仍为后续路线图。

2. 要解决的问题

以下问题陈述目前基于团队假设。开发前置确认（Week 0 并行，负责人：产品）：用 3–5 名目标学生做轻量访谈或演示反馈，确认核心痛点究竟是“不知道何时学”（规划问题）还是“知道却做不到”（执行力/拖延问题）。决策规则：若 ≥3/5 受访者主诉是执行/拖延而非“不知何时学”，则不改变是否开发（P0 仍作为桌面原型推进），但把演示与验收目标改为验证“桌面集中排程 + 周内重排”的可用性，而非证明产品市场需求或“影响执行行为”；并在路线图中把“执行力/签到”机制列为下一个待验证假设。

信息来源太分散

学生的任务分散在 Schoology、日历、教学大纲、老师公告、PDF、成绩册、体育训练、家庭安排和口头提醒里。

日历不会主动规划

普通日历能显示截止日期，但不会决定什么时候研究、起草、复习、拍摄、编辑，也不会在计划被打乱后自动重排。

普通提醒容易被忽略

学生已经收到太多通知，所以单纯的提醒很容易被无视。系统需要更像对话，有上下文，并且能适应用户习惯，才可能真正影响行为。

3. 目标与非目标

目标区分“MVP 必达”与“愿景方向”，与第 15 节的逐周交付物对齐，避免过度承诺。

目标 · MVP 必达

  • 创建一个主周日程，合并固定安排与学校任务。
  • 接入真实 Schoology 导入课程/作业/截止日期（官方 API 优先、浏览器插件兜底，成绩只读展示，见第 7 节）。
  • 支持手动录入或粘贴作业信息；保留内置 Schoology fixture 作为离线/降级与开发路径。
  • 根据作业描述和评分量规，把大型任务拆成更小的工作块。
  • 允许用户通过点击编辑面板创建、移动、删除和编辑时间块；自由拖拽/拉伸为路线图增强项（见第 15 节 MVP Gate）。
  • 允许用户用自然语言和智能体对话，例如“我周六下午 5–8 点要出去”。
  • 解释每个时间块为什么被安排在那里，并显示不确定性。

目标 · 愿景方向（探索/演示级）

  • 随着使用学习偏好，例如“我喜欢跑步后做数学”（MVP 仅做记录，不做可信学习闭环）。
  • 成绩感知优先级估算（MVP 以只读示例演示，不承诺真实计算，见第 9 节）。
  • 自由拖拽/拉伸、菜单栏快速录入、导入 Apple Calendar 忙闲信息。（macOS 本地通知为 P1 committed 项，非愿景级，见第 15 节）

第一版不做什么（非目标）

按“渠道 / 能力 / 依赖 / 用户范围 / 平台”分类：

  • 渠道：不做非官方的个人微信、Instagram、Snapchat 机器人集成（官方渠道属未来路线图，见第 16 节）。
  • 能力：不自动提交作业或修改成绩；不承诺精确预测 GPA（平均学分绩点）；不在没有用户确认的情况下做重大计划变更。
  • 依赖：MVP 接入真实 Schoology（官方 API 优先、插件兜底）；若学区未开放个人 API key 且插件受阻，则降级到 fixture/粘贴并作为交付风险（见第 17 节）。不建立外部日历网络连接。
  • 用户范围：不做家长端 / 老师端，不做多人协作与小组作业协同。
  • 平台：不做 SaaS、不做 Web 应用、不做移动 App；MVP 是个人版 macOS 桌面应用。

桌面端说明：MVP 明确选择 Mac 桌面形态，优先服务“坐在电脑前做作业和规划”的使用场景；移动端、云同步和多设备体验不进入本轮设计。

4. 高层架构

系统设计的关键是：这是一个个人版 Mac 桌面应用。所有持久化状态默认保存在本机（SQLite），智能体编排与日程工具以本地进程运行；未来外部集成以可选 adapter 接入，但不改变核心日程模型。技术澄清：“本地”指数据存储与工具执行在本机；拆分/解释仍需调用 LLM（默认远程，可配置本地模型），故非完全离线，无网络时降级为手动编辑。

5. 核心概念：时间块

日程由“时间块”（block）组成。每个承诺、作业步骤、学习安排和临时变动都变成一个带有元数据的时间块，智能体可以对它进行推理。周期性事件（上课、睡觉、训练）以重复规则表达其意图。DST 正确性：重复事件按本地墙钟锚定（RFC 5545 的本地 DTSTART + IANA 时区 + RRULE），展开为具体实例时用时区库换算为绝对时刻；只有真正一次性的块才以 UTC 存储。若以 UTC 绝对时刻锚定重复，会在夏令时切换后整体漂移一小时（如 8:00 的课变 7:00/9:00），必须避免。P0 范围收敛：P0 把固定周期承诺在规划窗口内展开为具体实例（本地墙钟），完整 RRULE 引擎与“按单次实例覆盖”（override）移入路线图（见第 15 节、第 12 节）。

6. 智能体循环

这个产品是具备自主行动能力的（agentic）：模型拥有工具、状态和维护日程的责任，而不只是回答问题。循环由 Codex 的工具调用机制承载，但每一步都要处理失败、冲突与并发。

观察：导入新作业、日历事件、成绩、用户消息和手动修改。外部导入内容一律标记为不可信输入（见第 13 节）。
理解：识别截止日期、工作量、难度、依赖关系、成绩影响和用户偏好。
规划：在尊重固定安排和时间限制的前提下创建或调整时间块。
校验与消解：在执行前做约束校验——检测时间重叠、是否超出可用时间、是否越过截止日期，并对冲突进行裁决（确定性规则优先，见第 8 节）。
解释：为每个安排附上原因和信心程度。
提问：当不确定性较高时，提出具体问题，而不是随便猜。
执行：通过本地 MCP 工具更新 SQLite 中的日程、提醒和操作日志。对重复提交使用幂等键，失败时保留 proposal 并允许用户重试或撤销。
学习：根据用户修改和完成情况改进之后的计划。

健壮性约定：①工具调用失败/超时按可重试性与幂等键处理（幂等去重表见第 12 节 IdempotencyRecord）；②“规划”与新的用户输入之间存在竞态时，以最新的用户意图为准并重新校验；③写入采用乐观并发控制，以 block 级 version 为主，仅在结构性变更时校验一个较窄的 schedule 版本，避免 schedule 根版本成为争用热点而产生误报冲突；冲突时拒绝并提示，而非静默覆盖；④batchReschedule 的 confirmationToken 绑定 proposal 捕获的受影响 block 版本 + TTL：确认时若版本已变则返回部分冲突并提示重生成，过期需重新生成，杜绝陈旧确认落到已变状态。

7. Schoology 集成策略

真实 Schoology 接入是 MVP 的核心价值（直接回应第 2 节“信息太分散”）。接入方式：官方 API 优先、浏览器插件兜底，两条路径归一到同一个 ImportAdapter 接口，产出同一套 Assignment/Block/ImportSource 模型，不改排程器与日程工具契约。fixture/粘贴保留为离线、降级与开发路径。

MVP · 首选

官方 API 集成

本地导入 adapter 用 Schoology 官方 REST API + OAuth 1.0a 读取课程、作业、截止日期、事件（成绩只读）。前提：学生能在账号设置生成个人 API key；Week 0 spike 先验证该前提是否成立。最稳、最结构化。

MVP · 兜底

浏览器插件导入

若学区未开放个人 API key，则走插件：学生正常登录 Schoology，插件读取学生已经能看到的页面并提取结构化数据。与 API 共用同一 ImportAdapter 接口；对页面改版更敏感、需维护。

开发/降级

fixture / 粘贴

内置 fixture、手动输入、粘贴教学大纲解析（LLM）。用于并行开发、离线演示，以及真实接入两条路径都受阻时的降级路径。解析结果进入 Assignment 草稿，用户确认后写入。

MVP 决策：真实 Schoology 进入 MVP（见第 15 节新增 Schoology 集成 Phase，工期扩到 4–5 周）。Week 0 前置 spike：用真实账号验证能否生成个人 API key 并成功拉一次 sections/{section_id}/assignments；成立→走 API，不成立→走插件，两者都不通→本轮降级到 fixture/粘贴并记为交付风险（见第 17 节）。所有外部数据先进入本地 Assignment 草稿，去重与来源标记后由用户确认写入 SQLite。

统一契约：官方 API、插件、fixture/粘贴、以及未来的 Apple Calendar 忙闲导入都实现同一 ImportAdapter，把外部数据转成同一套 Assignment / Block / ImportSource 模型；新增来源不改排程器和日程工具契约。Apple Calendar 忙闲导入仍为路线图。

Schoology 参考资料：Schoology 开发者文档包含 REST API、OAuth 1.0a（HMAC-SHA1 签名）认证、作业接口例如 sections/{section_id}/assignments，以及成绩、课程、用户相关接口。项目应避免收集 Schoology 密码；更稳妥的方式是官方 OAuth 或读取用户已经可见页面的插件导入。OAuth consumer key/secret 与 token 存 macOS Keychain，不落 SQLite 明文。

参考链接：Schoology API Documentation， Authentication， REST API。

健壮性：粘贴解析失败时保留原文并让用户手动补字段；导入 fixture 和外部 adapter 都必须做 schema 校验、来源标记和重复导入去重。

8. 日程智能

输入

  • 固定日程：睡觉、上课、运动、通勤、吃饭。
  • 学校数据：作业、截止日期、评分分类、成绩、评分量规。
  • 用户目标：“我想拿 4.0” 或 “我 AP 微积分 BC 要拿 A”。
  • 用户偏好：最佳专注时间、不喜欢的工作时间、学科顺序。
  • 现实变动：“我周六下午 5–8 点要出去”。

输出

  • 有开始和结束时间的日程块。
  • 大型作业和项目的子任务。
  • MVP 基于截止日期、工作量、固定安排和用户偏好的优先级；成绩影响仅作为只读演示，不进入真实排序。
  • 解释每个时间块为什么被安排在那里。
  • 当计划依赖未知信息时提出问题。

排程质量从哪里来（方向性说明）：这是产品的真正难点。MVP 采用“确定性约束 + LLM 分工”的混合方式：LLM 负责作业拆分与原因解释；确定性逻辑负责把子任务放进不冲突的可用时段（尊重固定块、截止日期、每日工作上限）。工作量估算来源优先级：用户输入 > 历史完成数据 > 从评分量规/作业类型推断的默认值。多约束冲突时按“临近截止日期 + 成绩影响 + 用户偏好”排序，并以人工确认兜底。注：MVP 阶段成绩计算未落地（见第 9 节），排序退化为“临近截止日期 + 用户偏好”，成绩影响待真实计算落地后接入。 MVP 接受“排得不完美”，但要求可解释、可撤销。

9. 成绩感知规划

成绩感知规划应该被视为估算，而不是保证。系统需要展示假设，让学生可以修正。MVP 中此功能以只读示例演示，不实现真实计算逻辑（见第 15 节），以控制范围与伦理风险。

例子

如果学生生物有 98，但 AP 微积分 BC 只有 91，而微积分考试又属于主要成绩项（major grade），智能体可以优先安排微积分复习，并解释原因：

我优先安排微积分复习，因为你目前接近 A 的分数线，而且这次考试属于主要成绩项（major）类别。

基线算法与保护措施

  • 基线启发式（未来真实计算方向，MVP 不实现）：优先级得分 = 分类权重 × 距目标成绩的差距 × 剩余可得分；权重来源须由用户确认或采用默认模板。
  • 数据缺失时给出不确定性区间，并标记为估算。
  • 让用户确认评分量规和分类权重。
  • 不要承诺精确的最终成绩。
  • 允许用户编辑每位老师的特殊规则。
  • 关注学生心理：避免以分数驱动过度挤压，必要时可关闭该功能。

10. 消息抽象层

智能体不应该直接操作 UI。Mac 版通过一个本地消息层统一处理应用内聊天、每日规划消息和 macOS 本地通知。

概念

智能体只调用一个内部函数，例如 sendMessage(message, options)。消息系统再决定显示在聊天流、提醒中心、菜单栏入口或 macOS Notification Center。

第一版实现与投递语义

  • P0 实现应用内聊天、手动触发的每日规划消息和提醒列表。
  • P1 使用 macOS 本地通知触发定时提醒（如“到微积分复习时间了”）。
  • 提醒必须去重并尊重安静时段 / 每日上限，避免第 2 节所说的“提醒轰炸”。
  • MVP 不接短信、邮件、微信、Instagram 或任何云消息渠道。

提醒执行模型（P1 起）：提醒任务以 Reminder 实体持久化，包含 dueAt、status、idempotencyKey 和关联 blockId。应用启动时从 SQLite 恢复待触发提醒，并通过 macOS 本地通知或应用内提醒展示；不依赖云队列、后台服务器或浏览器推送。

12. 数据模型

P0 推荐使用 SQLite（启用 WAL 与 busy_timeout），数据库文件保存在 macOS 应用数据目录。所有核心表默认包含 id、created_at、updated_at、deleted_at（软删除）和 version。时间存储：一次性块用 UTC epoch milliseconds 或 ISO-8601 UTC 字符串保存；重复块按本地 DTSTART + IANA 时区 + RRULE 存储（DST 正确性见第 5 节），展开为实例时换算；所有块单独保存 IANA timezone，显示时按用户时区转换。并发控制以 version int 为基础；未来导入/同步以 source、external_id、import_batch_id 去重。

13. 智能体可控性

智能体会自动修改本地日程，因此可控性与可信任是产品能被采纳的前提。本节关注“什么时候能自动提交、什么时候必须让用户确认、如何撤销和排查”。

可控性原则

  • 重大日程变更需要用户确认。
  • 为生成计划显示信心程度和原因。
  • 保留撤销历史（由 ActionLog 支撑）：P0 支持单步撤销上一批智能体操作；撤销为尽力恢复——若受影响块已被用户再次编辑，则返回冲突并要求人工选择（与第 11 节 undo() 契约一致），不静默覆盖。
  • 记录智能体操作，便于调试和建立信任。
  • MVP 不让智能体提交作业或给老师发消息。
  • 排程硬约束：不得压缩睡眠块，不得绕过固定块，不替用户完成或提交作业。

输入与工具约束

  • 作业描述、PDF 文本和粘贴内容只作为资料字段处理，不作为系统指令执行。
  • 高影响工具（删除、批量重排、移动固定块）强制确认（与本节“操作分级”表一致）。
  • 对工具输出做校验（时间合法、无越界），异常时回退到人工确认。
  • 为智能体决策保留可观测性（日志/追踪），便于回归与排错。
  • 工具提交由本地策略和版本校验决定，不由模型自由改 SQLite。

14. 成功指标与验收标准

没有可衡量的成功标准，项目结束时就无法判断成败。以下区分“原型验证指标”与“未来产品北向指标”。

原型（MVP）验收指标

  • 以下 [GATE] 项为 go/no-go 硬门槛（全过才判 go），[OBS] 项为观测项不阻塞。
  • [GATE] 演示端到端：第 18 节演示脚本可无开发者干预跑通（含关闭重开持久化、批量重排确认、撤销恢复）；允许正常用户确认 proposal 与高风险变更。
  • [GATE] 排程质量：在 ≥3 个真实一周作业 fixture 上，生成计划0 处硬约束违规（不压睡眠/不越截止/不逆依赖），且经 1 名目标学生评为“可用”（可小改）。
  • [GATE] 真实 Schoology 导入：用真实账号经官方 API 或插件路径之一，拉到本周真实作业与截止日期，映射为 Assignment/Block 并按 external_id 去重、标记来源；重复同步不产生重复块。例外：若 Week 0 spike 确认 API 与插件两条真实路径均不可用，本条降级为 fixture 演示并在 Gate 明确记录（此时不判 fail，但视为核心价值未验证，见第 17 节）。
  • [GATE] 拆分质量：评测集 N≥20 fixture，项目作业拆分人工通过率 ≥80%（“通过”＝按§8 rubric 检查表由指定评审判定：含研究/制作/复核类可执行块且工时估算合理）。
  • [OBS] 接受率（北极星雏形）：AI 生成时间块被用户直接接受（未重排/删除）的比例，原型期观测参考 ≥60%，观测项而非门槛。
  • [GATE] 数据持久化：录入的作业与时间块关闭重开后不丢失。
  • [OBS] 冷启动成本：由首次使用的非开发者在目标 Mac 上计时——固定安排设置 与 “录入 3 个作业”分别计时（合计参考 ≤8 分钟，作观测非门槛）；粘贴教学大纲可提取标题、截止日期和粗略工时。
  • [GATE] 桌面可用性：Mac 应用可冷启动、关闭后重开、离线查看既有日程；SQLite 数据文件可导出或清空。
  • [GATE] 可控性门槛：批量重排先显示 proposal，确认后写入；撤销为单步、尽力恢复上一批智能体操作（受影响块已被再次编辑时返回冲突需人工选择，见第 11 节/第 13 节）。

未来产品北向指标

  • 采纳与留存（如次周留存）。
  • 计划被接受 vs 被修改的比例（衡量排程质量）。
  • 行为影响：按计划完成的时间块比例（衡量是否真的影响行为）。
  • 拆分与工时估算的准确度随使用提升。

15. 四至五周 MVP 计划

技术底座：个人版 Mac 桌面应用，本地运行 Codex 工具调用循环与 MCP Schedule Server，SQLite 保存日程数据。聚焦单用户本地可信闭环 + 真实 Schoology 导入：时间块模型、Mac 周日程界面、真实 Schoology（官方 API 优先/插件兜底）与手动/粘贴/fixture 导入、MVP 工具契约、确定性排程、proposal 确认与撤销。工期：基准 4 周，Week 5 为缓冲/应急（尤其当 API key 不可用、需走插件路径时启用）。人力假设：2 名全职原型开发者（基准计划按 2 人排期）；若只有 1 人，按下方“1 人降级范围”执行——去掉 batchReschedule 与聊天改期，只保留单块编辑 + 单作业拆分排程 + 单步撤销，Schoology 只做 API 只读导入。范围策略：先做透 Mac P0 + 真实 Schoology 作业/截止日期导入；云同步、多用户、外部日历、成绩计算、偏好学习闭环、完整 RRULE 重复引擎 + 按实例覆盖 + DST 全量处理、批量撤销、自由拖拽均移入路线图。

Week 0 开发启动门槛（≤3 天，spike 通过才进 Week 1）：①跑通本地 Codex → 单一 MCP server → 工具调用 spike，验证 getSchedule() 往返；②Schoology 接入 spike：用真实账号验证能否生成个人 API key 并成功拉一次 sections/{section_id}/assignments——成立→走官方 API，不成立→评估插件可行性，两者都不通→本轮降级 fixture/粘贴并记为交付风险（见第 7 节、第 17 节）。Fallback：若 Codex 集成受阻，P0 降级为“瘦编排层直连 LLM tool-calling API + 同一套 MCP 工具”，不阻塞交付。

启动前须锁定的决策（Decisions to lock before Day 1）： ①桌面壳＝TypeScript + Tauri（已锁定；含 rrule.js/Luxon 处理重复与时区，跨平台为未来可选、P0 仅 Mac 验收）； ②LLM 提供方/模型 + token 预算数值 + 访问已开通； ③SQLite 访问层方案； ④MCP 工具 schema 草案（第 11 节）与单一常驻 server 拓扑（第 4 节）； ⑤Schoology 接入方式＝官方 API 优先、插件兜底（已锁定；Week 0 spike 定夺具体路径）； ⑥第 2 节用户访谈的负责人与时间（在 Week 0 并行完成，决策规则见该节）。

Week 1 · P0

日程基础

  • 创建 Mac 桌面壳、周日程主窗口、作业录入面板和聊天侧栏。
  • 建立 SQLite schema 与迁移：LocalProfile、Schedule、Block、Assignment、Message、ActionLog、ImportSource。
  • 跑通 getSchedule() 与 createBlock(dryRun) 本地 MCP 工具。
  • 创建固定周日程设置（周期承诺在规划窗口内展开为具体实例，本地墙钟，不做完整 RRULE 引擎）；支持点击编辑面板修改时间块（自由拖拽为路线图项）。
  • 手动作业输入与内置 Schoology fixture（真实 Schoology 接入排入 Week 3）；把所有任务表示为时间块并持久化。粘贴解析依赖 LLM，排入 Week 2。

DoD：用户可手动录入 5 个作业、在周视图看到对应时间块并能通过点击面板编辑其时间；关闭应用再打开后数据仍在。

Week 2 · P0

智能体规划

  • 以 MCP 工具注册创建/移动/删除时间块的日程工具。
  • 把大型作业拆成子任务，附原因说明与信心标签；粘贴教学大纲解析（LLM）进入本地 Assignment 草稿。
  • 实现确定性 scheduler（依赖按拓扑序为硬约束）、冲突检测与 batchReschedule(dryRun) proposal。
  • 添加通过聊天修改日程的命令；时间块完成态与未完成顺延（done/partial/incomplete，completeBlock）；为 LLM 规划建立评测集（N≥20 fixture）。

DoD：给定 1 个作业，端到端“拆分→排入日程→解释理由”；能响应一次自然语言改期；标记一个块未完成可把剩余工作顺延重排；新增紧急作业可生成批量重排 proposal，用户确认后原子提交并可单步撤销。

Week 3 · P0

真实 Schoology 集成

  • 实现统一 ImportAdapter 接口；按 Week 0 spike 结论落地官方 API（OAuth 1.0a）路径，读取课程/作业/截止日期（成绩只读），token 存 Keychain。
  • 字段映射到 Assignment/Block/ImportSource；按 external_id 去重与来源标记；错误/超时/部分失败处理与用户确认写入。
  • 若 API key 不可用则实现浏览器插件兜底路径（读已登录页面），共用同一 adapter 契约（可能顺延到 Week 5）。
  • 真实作业进入既有拆分/排程闭环；重复同步不产生重复块。

DoD：用真实账号拉到本周真实作业与截止日期，映射为时间块、去重、可解释；断网/接口失败时降级 fixture 且不崩溃。

Week 4 · P1 + 收尾

集成、测试与演示

  • P0 收尾：单步撤销、错误状态、空闲时间不足、版本冲突、Mac 窗口尺寸适配。
  • 预留 20–30% 时间做测试与修 bug（不可挤占——超期时先砍下面 stretch 项，不砍测试）；打磨 UI 与演示流程；含 Schoology 导入回归。
  • 演示包与分发：准备可运行的本地 Mac 演示包与限制说明；提前完成签名/公证（notarization），并在目标 Mac 上验证 Gatekeeper 放行。
  • Stretch（P0 稳定后才做，可延后）：P1 macOS 本地通知预览；成绩感知优先级只读示例（不实现真实计算，明确标注）。

DoD：演示脚本全部跑通（含真实 Schoology 导入、关闭重开持久化与撤销恢复）；关键路径有回归用例；智能体错误排程可单步撤销恢复；已签名/公证的应用可在目标 Mac 上本地启动。

Week 5 · 缓冲/应急

缓冲周（按需启用）

  • 吸收上游超期（尤其 Codex spike、Schoology 认证/字段映射、页面改版适配）。
  • API key 不可用时，在此完成浏览器插件兜底路径与测试。
  • 补测试、修 bug、打磨演示；若一切顺利可提前结项或试做 P1 通知。

DoD：Week 4 未闭合项在此收敛；真实 Schoology（API 或插件之一）导入达标或已在 Gate 记录为不可用降级。

18. 演示脚本

  1. 输入固定安排：睡觉、上课、田径、健身、吃饭。
  2. （onboarding）计时冷启动：由首次使用的非开发者在目标 Mac 上，分别计时固定安排设置与录入 3 个作业，并演示粘贴解析如何降低录入成本。
  3. 真实 Schoology 导入：用真实账号（官方 API 或插件路径）拉取本周真实作业与截止日期，展示映射为时间块、去重与来源标记；接口失败时演示降级到 fixture。
  4. 导入或创建一个有截止日期和评分量规的生物纪录片作业（可用 Schoology 真实作业或内置示例）。
  5. 让智能体规划这个作业，展示把作业拆成规划、研究、拍摄、剪辑和复习时间块，并说明每个块为什么被安排在那里。
  6. 压力场景：临时新增一个紧急作业，展示智能体触发批量重排 proposal，用户确认后原子提交以消解冲突。
  7. 撤销：对上一步批量重排执行 undo，验证日程恢复到重排前状态。
  8. 告诉智能体：“我周六下午 5–8 点要出去。”展示日程更新，并让智能体解释移动了什么。
  9. 完成态：标记某个学习块“只完成一半”，展示剩余工时顺延重排。
  10. 持久化：关闭应用并重新打开，验证作业、时间块与修改仍在。
  11. 展示“若无此产品学生会怎样”的对照，说明 AI 排布的取舍。
  12. 展示每日规划消息，以及成绩感知优先级示例（明确标注为“未来能力预览 / 只读示例”，不计入验收门槛）。

参考项目（Drive Master），基于 Codex 二次开发；本项目仅参考其聊天 UI 与工具编排模式，不复用其 Web/OAuth/多用户形态。
时间块 / block
日程的基本单元，带元数据、可被智能体推理与移动。
评分量规 / rubric
老师给出的评分标准，说明各部分如何计分。
主要成绩项 / major grade
权重较高的成绩项（如大考、大项目）。
GPA
平均学分绩点（Grade Point Average）。
AP 微积分 BC
美国大学先修（Advanced Placement）微积分 BC 课程。
agentic
具备自主行动能力的——模型拥有工具、状态与目标，而不只是应答。
RRULE
日历重复规则（Recurrence Rule），用于表示周期性事件。
OAuth 1.0a
Schoology 采用的开放授权协议版本（HMAC-SHA1 签名）。
REST API
基于 HTTP 的应用编程接口风格。
Prompt injection
攻击者通过不可信输入操纵 LLM 执行非预期行为。
ToS
服务条款（Terms of Service）。
ImportAdapter
统一的外部数据导入接口；官方 API、浏览器插件、fixture/粘贴均实现它，产出同一套 Assignment/Block/ImportSource 模型。
DoD
完成定义（Definition of Done），任务被视为完成的可验证标准。

讨论与迭代用 HLDD 草稿。推荐 MVP 是一个个人版 Mac 桌面应用：本地日程板加智能体循环，能拆分作业、安排时间块、解释决策，并响应自然语言的日程变化。
