1. 产品愿景

概述

这个产品是一个面向学生的 AI 日程智能体。它把学校作业、成绩、固定安排、个人偏好以及临时变动整合进一个主日程。它不只是展示任务列表，而是主动推理工作应该安排在什么时候，解释原因，在不确定时提问，并根据用户调整逐渐学习。

第一版专注于学校场景，尤其是与 Schoology 相关的工作流。长期来看，它可以扩展成任何人都能使用的任务与时间规划助手（迁移路径见第 16 节）。

一句话定位

一个能把学校平台、作业、评分量规（rubric）、成绩和个人生活安排转化为动态日程，并能通过聊天修改计划的智能体。

2. 要解决的问题

以下问题陈述目前基于团队假设，尚未经过系统性用户研究验证。MVP 前置动作：对 5–10 名目标高中生做轻量访谈，确认核心痛点究竟是“不知道何时学”（规划问题）还是“知道却做不到”（执行力/拖延问题）——二者对应的产品形态差异很大。

信息来源太分散

学生的任务分散在 Schoology、日历、教学大纲、老师公告、PDF、成绩册、体育训练、家庭安排和口头提醒里。

日历不会主动规划

普通日历能显示截止日期，但不会决定什么时候研究、起草、复习、拍摄、编辑，也不会在计划被打乱后自动重排。

普通提醒容易被忽略

学生已经收到太多通知，所以单纯的提醒很容易被无视。系统需要更像对话，有上下文，并且能适应用户习惯，才可能真正影响行为。

3. 目标与非目标

目标区分“MVP 必达”与“愿景方向”，与第 15 节的逐周交付物对齐，避免过度承诺。

目标 · MVP 必达

  • 创建一个主周日程，合并固定安排与学校任务。
  • 支持手动录入或粘贴作业信息；提供模拟 Schoology 数据。
  • 根据作业描述和评分量规，把大型任务拆成更小的工作块。
  • 允许用户手动拖动、拉伸、删除和编辑时间块。
  • 允许用户用自然语言和智能体对话，例如“我周六下午 5–8 点要出去”。
  • 解释每个时间块为什么被安排在那里，并显示不确定性。

目标 · 愿景方向（探索/演示级）

  • 随着使用学习偏好，例如“我喜欢跑步后做数学”（MVP 仅做记录，不做可信学习闭环）。
  • 成绩感知优先级估算（MVP 以只读示例演示，不承诺真实计算，见第 9 节）。
  • 在可行时从 Schoology 官方 API 导入作业、截止日期、活动和成绩。

第一版不做什么（非目标）

按“渠道 / 能力 / 依赖 / 用户范围 / 平台”分类：

  • 渠道：不做非官方的个人微信、Instagram、Snapchat 机器人集成（官方渠道属未来路线图，见第 16 节）。
  • 能力：不自动提交作业或修改成绩；不承诺精确预测 GPA（平均学分绩点）；不在没有用户确认的情况下做重大计划变更。
  • 依赖：MVP 不依赖学校管理员批准 API 权限，也不依赖浏览器插件抓取。
  • 用户范围：不做家长端 / 老师端，不做多人协作与小组作业协同。
  • 平台：不做移动原生 App；MVP 为网页应用（移动端策略见下方说明）。

移动端说明：高中生是移动优先人群，桌面网页形态可能与实际使用环境错配。MVP 采用响应式网页以在手机浏览器可用，原生 App 列入路线图；此风险在第 17 节登记。

系统设计的关键是：智能体可以接入多个数据来源和多个消息渠道，但核心日程模型保持稳定。下图三层之间为数据/调用流向。

  • 应用内聊天
  • 拖拽式日历界面
  • 应用内提醒（MVP）
  • 可选浏览器插件（路线图）

后端与智能体层

推理与编排发生的地方。

  • Codex app-server（智能体运行时 + 工具调用循环）

  • LLM 推理服务（模型、上下文/记忆、成本预算）
  • 日程状态存储（持久化 + 版本/乐观并发）
  • 调度引擎（定时提醒、重排触发）
  • 日程工具 / Schoology 导入工具 / 偏好记忆

→

集成层

应用可以读取或写入的外部系统。

  • Schoology 官方 API（OAuth 1.0）或插件导入
  • Apple / Google Calendar（双向同步）
  • 未来的邮件或短信
  • 未来的官方微信或其他消息平台
  • 数据库

技术底座与复用边界：智能体运行时基于最新上游 OpenAI Codex（openai/codex，含 Rust 核心 codex-rs、Codex SDK 与 app-server），仅复用其“LLM → 工具 → 结果”的工具调用循环机制；聊天 UI 与 OAuth 集成复用 abg-agent-swarm（Drive Master）的现成实现与模式。日程相关的自定义工具经 MCP server 暴露给 Codex 调用，尽量不 fork codex-rs 核心。

需要说明的是，Codex 本质是单用户编码智能体，因此多用户会话隔离、领域数据持久化、调度引擎（定时提醒）属于本项目的净新增工作量，非底座自带。本项目还需新增学校日程特定部分：时间块模型、拖拽日历界面、Schoology 导入工具与日程工具。底座省去的是运行时循环与前端骨架，但上述封装仍需投入（见第 17 节风险）。

非功能性方向（原型阶段的量级假设）：单区域部署，支持数百名试点用户；为 LLM 调用设定 token 预算与缓存策略（见第 17 节成本风险）；外部日历双向同步须定义权威源与冲突解决策略（后端改动 vs 外部改动以谁为准）。

5. 核心概念：时间块

日程由“时间块”（block）组成。每个承诺、作业步骤、学习安排和临时变动都变成一个带有元数据的时间块，智能体可以对它进行推理。周期性事件（上课、睡觉、训练）通过重复规则（RRULE）表示，而非逐次复制。

时间块类型
时间块类型 例子 是否可移动 来源
固定时间块 上课、睡觉、田径训练、家庭活动 通常不可移动 用户、日历、Schoology
导入的截止日期 生物纪录片周五截止 截止日期固定，工作时间可移动 Schoology、插件、手动导入
生成的工作块 研究主题、拍摄采访、剪辑视频 可移动 智能体
灵活习惯块 健身、阅读、复习、补作业 可移动 用户或智能体
临时变动 “周六下午 5–8 点出去玩” 通常固定 用户聊天

6. 智能体循环

这个产品是具备自主行动能力的（agentic）：模型拥有工具、状态和维护日程的责任，而不只是回答问题。循环由 Codex 的工具调用机制承载，但每一步都要处理失败、冲突与并发。

观察：导入新作业、日历事件、成绩、用户消息和手动修改。外部导入内容一律标记为不可信输入（见第 13 节）。
理解：识别截止日期、工作量、难度、依赖关系、成绩影响和用户偏好。
规划：在尊重固定安排和时间限制的前提下创建或调整时间块。
校验与消解：在执行前做约束校验——检测时间重叠、是否超出可用时间、是否越过截止日期，并对冲突进行裁决（确定性规则优先，见第 8 节）。
解释：为每个安排附上原因和信心程度。
提问：当不确定性较高时，提出具体问题，而不是随便猜。
执行：更新日程、提醒和可选的日历输出。对外部写入使用幂等键（重复执行不产生重复事件），失败时重试并可回退。
学习：根据用户修改和完成情况改进之后的计划。

健壮性约定：①工具调用失败/超时按可重试性与幂等键处理；②“规划”与新的用户输入之间存在竞态时，以最新的用户意图为准并重新校验；③写入采用乐观并发控制（基于版本号），冲突时拒绝并提示，而非静默覆盖（数据模型见第 12 节）。

7. Schoology 集成策略

MVP 模式

手动或模拟导入

使用样例数据或用户粘贴的作业信息（可让智能体解析粘贴的教学大纲以降低录入摩擦）。这样即使还没有学校管理员权限，也可以先验证日程和智能体逻辑。

原型模式（路线图）

浏览器插件导入

学生正常登录 Schoology，插件只读取学生已经能看到的页面内容并提取结构化数据。该路径受服务条款（ToS）限制，须先做评审，非 MVP 范围。

生产模式

官方 API 集成

如果学校批准访问，后端使用 Schoology 官方 API 和 OAuth 1.0 流程，更稳定地读取课程、作业、成绩、事件和提交信息。

Schoology 开发者文档包含 REST API、OAuth 1.0 认证、作业接口例如 sections/{section_id}/assignments，以及成绩、课程、用户相关接口。

OAuth 1.0 说明：Schoology 使用 OAuth 1.0（HMAC-SHA1 签名），已被业界普遍以 OAuth 2.0 取代。consumer key/secret 与 token 存放在服务端、绝不下发到客户端或插件，并定义撤销与轮换机制。abg-agent-swarm 的 Drive OAuth 属 OAuth 2.0，Calendar 授权可较直接复用其模式；Schoology 的 OAuth 1.0 需另实现签名流程，仅复用 token 服务端保管/撤销/轮换这套结构。

健壮性：插件抓取须对选择器容错、对返回数据做 schema 校验、解析失败时降级为手动导入；API 集成须处理速率限制（指数退避、增量同步/ETag、合理轮询间隔）。

参考链接：Schoology API Documentation， Authentication， REST API。

8. 日程智能

输入

  • 固定日程：睡觉、上课、运动、通勤、吃饭。
  • 学校数据：作业、截止日期、评分分类、成绩、评分量规。
  • 用户目标：“我想拿 4.0” 或 “我 AP 微积分 BC 要拿 A”。
  • 用户偏好：最佳专注时间、不喜欢的工作时间、学科顺序。
  • 现实变动：“我周六下午 5–8 点要出去”。

输出

  • 有开始和结束时间的日程块。
  • 大型作业和项目的子任务。
  • 基于紧急程度、工作量和成绩影响的优先级。
  • 解释每个时间块为什么被安排在那里。
  • 当计划依赖未知信息时提出问题。

排程质量从哪里来（方向性说明）：这是产品的真正难点。MVP 采用“确定性约束 + LLM 分工”的混合方式：LLM 负责作业拆分与原因解释；确定性逻辑负责把子任务放进不冲突的可用时段（尊重固定块、截止日期、每日工作上限）。工作量估算来源优先级：用户输入 > 历史完成数据 > 从评分量规/作业类型推断的默认值。多约束冲突时按“临近截止日期 + 成绩影响 + 用户偏好”排序，并以人工确认兜底。注：MVP 阶段成绩计算未落地（见第 9 节），排序退化为“临近截止日期 + 用户偏好”，成绩影响待真实计算落地后接入。 MVP 接受“排得不完美”，但要求可解释、可撤销。

9. 成绩感知规划

成绩感知规划应该被视为估算，而不是保证。系统需要展示假设，让学生可以修正。MVP 中此功能以只读示例演示，不实现真实计算逻辑（见第 15 节），以控制范围与伦理风险。

例子

如果学生生物有 98，但 AP 微积分 BC 只有 91，而微积分考试又属于主要成绩项（major grade），智能体可以优先安排微积分复习，并解释原因：

我优先安排微积分复习，因为你目前接近 A 的分数线，而且这次考试属于主要成绩项（major）类别。

基线算法与保护措施

  • 基线启发式（未来真实计算方向，MVP 不实现）：优先级得分 = 分类权重 × 距目标成绩的差距 × 剩余可得分；权重来源须由用户确认或采用默认模板。
  • 数据缺失时给出不确定性区间，并标记为估算。
  • 让用户确认评分量规和分类权重。
  • 不要承诺精确的最终成绩。
  • 允许用户编辑每位老师的特殊规则。
  • 关注学生心理：避免以分数驱动过度挤压，必要时可关闭该功能。

10. 消息抽象层

智能体不应该直接绑定某一个平台。它应该通过一个消息层发送提醒，这样原型现在能工作，将来也能添加更多渠道。

概念

智能体只调用一个内部函数，例如 sendMessage(user, message, options)。消息系统再决定通过应用内聊天、应用内提醒、邮件、短信、官方微信或其他未来渠道发送。

第一版实现与投递语义

  • 先实现应用内聊天与应用内提醒（MVP 不依赖浏览器推送）。
  • 由调度引擎触发定时提醒（如“到微积分复习时间了”）。
  • 定义投递语义：失败重试、渠道降级（fallback）、去重与频率控制（安静时段 / 每日上限），避免第 2 节所说的“提醒轰炸”。
  • 接口预留短信、官方微信或邮件的接入位置。
  • MVP 不使用非官方的个人微信或 Instagram 机器人。

11. 智能体工具

下表为工具概览。工具经 MCP server 暴露给 Codex 调用（不 fork codex-rs 核心）。每个工具在实现前须补全契约：输入/输出 schema、错误码、是否需用户确认、是否幂等。带副作用的工具（写日历、发消息、批量重排）默认需要确认。

智能体工具
工具 用途 例子
getSchedule() 读取当前日程块和限制。 寻找周五前的空闲时间。
getAssignments() 查询作业、截止日期与状态。 列出本周到期的所有作业。
createBlock() 添加学习块或固定事件。 添加周二 19:00–20:30 的微积分复习。
moveBlock() 在冲突或用户请求后重排时间块。 把生物研究移动到晚饭后。
deleteBlock() 删除时间块。 删除已取消的训练。
batchReschedule() 作为单一原子操作批量重排多个块。 临时新增紧急作业后一次性重排本周。
splitAssignment() 把大型作业拆成子任务。 把纪录片拆成规划、研究、拍摄、剪辑。
setPreference() 存储从聊天或编辑中学到的偏好。 偏好跑步后做数学。
askClarifyingQuestion() 在不确定性太高时提问。 询问拍摄需要 2 小时还是 4 小时。
undo() 撤销上一步智能体操作（依赖操作历史）。 撤销刚才的自动重排。
sendMessage() 通过消息层通知或对话。 “到微积分复习时间了，还按原计划吗？”

13. 智能体安全与可控性

智能体会自动修改用户日程，因此可控性与可信任是产品能被采纳的前提。本节关注智能体的行为安全与可控性。

可控性原则

  • 重大日程变更需要用户确认。
  • 为生成计划显示信心程度和原因。
  • 保留撤销历史（由 ActionLog 支撑），支持一键回退。
  • 记录智能体操作，便于调试和建立信任。
  • MVP 不让智能体提交作业或给老师发消息。

输入与工具防护

  • 作业描述、抓取页面等外部内容为不可信输入，与指令隔离（结构化封装/分隔），防止 prompt injection 操纵智能体。
  • 高副作用工具（sendMessage、写日历、批量重排）强制确认与权限边界。
  • 对工具输出做校验（时间合法、无越界），异常时回退到人工确认。
  • 为智能体决策保留可观测性（日志/追踪），便于回归与排错。

14. 成功指标与验收标准

没有可衡量的成功标准，项目结束时就无法判断成败。以下区分“原型验证指标”与“未来产品北向指标”。

原型（MVP）验收指标

  • 硬性门槛：第 18 节演示脚本可端到端无人工干预跑通。
  • 拆分质量：给定一个带评分量规的作业，智能体拆出 ≥4 个合理子任务且时间估算被用户认可。
  • 接受率（北极星雏形）：AI 生成的时间块中被用户直接接受（未重排/删除）的比例，原型期观测参考 ≥60%，作为观测项而非 go/no-go 门槛。
  • 数据持久化：录入的作业与时间块刷新后不丢失。

未来产品北向指标

  • 采纳与留存（如次周留存）。
  • 计划被接受 vs 被修改的比例（衡量排程质量）。
  • 行为影响：按计划完成的时间块比例（衡量是否真的影响行为）。
  • 拆分与工时估算的准确度随使用提升。

15. 三周 MVP 计划

技术底座：智能体运行时基于上游 openai/codex（复用工具调用循环），聊天 UI 与 OAuth 复用 abg-agent-swarm 现成实现；因此三周聚焦于学校日程特定功能（时间块模型、拖拽日历、Schoology 导入、日程工具与排程逻辑）以及多用户封装 / 持久化 / 调度引擎等净新增工作，范围压力较从零开发显著降低。人力假设：1–2 名全职原型开发者。范围策略：先做透一条最小可信循环（P0），其余标注为 P1/Stretch 或移入路线图；每周给出可验证的完成定义（DoD）。

Week 1 · P0

日程基础

  • 在 Codex 底座上接好聊天/后端骨架，跑通一个自定义 MCP 工具被 Codex 调用（底座验证点）。
  • 创建响应式网页日程板与固定周日程设置；支持点击编辑面板修改时间块（自由拖拽为增强项）。
  • 手动作业输入或模拟 Schoology 数据（锁定此路径，不做插件）；把所有任务表示为时间块并持久化。

DoD：跑通一个 MCP 工具调用；用户可录入 5 个作业、在周视图看到对应时间块并能编辑其时间；刷新后数据仍在。

Week 2 · P0

智能体规划

  • 以 MCP 工具注册创建/移动/删除时间块的日程工具。
  • 把大型作业拆成子任务，附原因说明与信心标签。
  • 实现冲突检测与原子批量重排（batchReschedule）。
  • 添加通过聊天修改日程的命令；为 LLM 规划建立小型评测集（若干标准案例）。

DoD：给定 1 个作业，端到端“拆分→排入日程→解释理由”，能响应一次自然语言改期；新增一个紧急作业能触发一次成功的批量重排消解冲突。

Week 3 · P1 + 收尾

集成、测试与演示

  • 应用内提醒（浏览器推送移入路线图）。
  • 成绩感知优先级：只读示例（不实现真实计算）。
  • 预留 20–30% 时间做测试与修 bug；打磨 UI 与演示流程。
  • 准备最终展示与限制说明。

DoD：演示脚本全部跑通；关键路径有回归用例；智能体错误排程可撤销恢复。

MVP Gate（go / no-go）：三周结束时按第 14 节验收指标评估，决定是否继续、进入路线图哪个阶段。降级/移出 MVP 的项：浏览器通知、真实成绩感知计算、偏好学习闭环、Schoology 插件与官方 API。

16. 长期路线图

Schoology API

在学校批准后，使用官方 OAuth/API 集成。

浏览器插件

在完成 ToS 评审后，为无管理员集成的用户导入可见 Schoology 数据。

17. 风险与缓解

风险登记册
风险 影响 缓解方式
范围管理（3 周） 仍需在期限内交付；日程特定部分（拖拽日历、拆分质量、排程逻辑）是主要工作量。 复用 abg-agent-swarm（Codex）底座省去运行时/聊天 UI/OAuth 从零开发；锁定 P0 最小循环，其余移入路线图。
底座契合度 / 适配成本 Codex 是单用户编码智能体，包装成多用户 + 定时提醒的 Web 服务需额外适配；熟悉其构建与 MCP 工具注册也有学习曲线。 日程工具走 MCP、不 fork 核心；Week 1 设底座验证点；把多用户封装/持久化/调度作为净新增显式排期。
后台调度可靠性 定时提醒需在进程重启后仍可触发，否则提醒丢失。 使用成熟定时/队列组件，提醒设计为幂等可重放。
并发写冲突 用户拖拽与智能体自动重排同时发生可能互相覆盖。 基于版本号的乐观并发控制，冲突时拒绝并提示，而非静默覆盖。
移动端形态错配 桌面网页与高中生移动优先习惯错配，影响采纳。 MVP 采用响应式网页保证手机可用；原生 App 列入路线图。
LLM 成本与延迟 每次循环都调用 LLM，成本/延迟可能失控。 设定 token 预算上限、缓存与最小上下文；确定性逻辑承担排程，LLM 只做拆分与解释。
Schoology 数据获取失败 手动录入削弱“信息聚合”价值；插件受 DOM/反爬/ToS 影响失效。 MVP 锁定手动+模拟并降低录入摩擦（解析教学大纲）；插件/API 作为需评审的升级路径。
智能体做出差的日程安排 错误计划会很快降低用户信任。 显示原因、信心、撤销，要求重大变更确认；确定性约束兜底 + LLM 评测集。
提醒被忽略 产品必须真的影响行为。 用签到式对话、每日规划固定环节，以及未来更高可见度的渠道；控制频率避免轰炸。
成绩估算错误 学生可能过度信任成绩计算。 展示假设、标记为估算；MVP 仅做只读示例。
无 QA / 回滚 自动改日程的产品缺回归测试尤其危险。 Week 3 预留测试缓冲，建关键路径回归用例，提供撤销/恢复。

18. 演示脚本

  1. 输入固定安排：睡觉、上课、田径、健身、吃饭。
  2. （onboarding）展示冷启动录入的摩擦评估：录入课表与固定安排大约需要多久，如何用粘贴解析降低成本。
  3. 导入或创建一个有截止日期和评分量规的生物纪录片作业。
  4. 让智能体规划这个作业，展示把作业拆成规划、研究、拍摄、剪辑和复习时间块，并说明每个块为什么被安排在那里。
  5. 压力场景：临时新增一个紧急作业，展示智能体触发批量重排以消解冲突。
  6. 告诉智能体：“我周六下午 5–8 点要出去。”展示日程更新，并让智能体解释移动了什么。
  7. 展示“若无此产品学生会怎样”的对照，说明 AI 排布的取舍。
  8. 展示每日规划消息，以及成绩感知优先级示例（明确标注为“未来能力预览 / 只读示例”）。
