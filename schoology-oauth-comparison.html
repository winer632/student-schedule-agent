Schoology API Authentication

Three-Legged OAuth 与 Two-Legged OAuth 的应用场景区别

两者的关键区别不是“哪个能调用更多接口”，而是每次 API 请求到底代表谁： Three-Legged OAuth 代表某个授权用户，Two-Legged OAuth 代表 API key 所属的账号、管理员或机构集成身份。 对学生个人自用的本地 Mac 桌面应用，优先验证 Two-Legged / Personal API key 是否可用。

核心结论

Schoology 的 OAuth 1.0 认证把“应用身份”和“用户权限”拆开处理。 consumer_key 识别应用或 API 使用方；在三方 OAuth 中， oauth_token 进一步识别具体授权用户。

所以实际能访问哪些学生、课程、班级、成绩或用户资料，通常取决于被代表的用户在 Schoology 中的角色、学校、楼宇和课程上下文。

一句话区分

Three-Legged

用户授权型。适合第三方应用代表当前登录用户访问其有权限访问的数据。

Two-Legged

系统集成型。适合学校、管理员或后台系统代表自己访问 API。

当前产品形态首选：Two-Legged / Personal API key

学生个人 Mac 桌面应用怎么选

如果应用是学生自己开发、自己在本机使用的 macOS 桌面应用，并且目标只是读取自己的课程、作业、截止日期和成绩只读信息， 那么它更接近“个人 API 使用方访问自己的数据”，不是“一个第三方 SaaS 代表很多用户访问数据”。

因此首选路径是先验证学生账号是否能获取个人 API credentials，然后用 Two-Legged OAuth 调用 Schoology API。 这个判断的前提是学校或学区允许该学生角色使用 Access Schoology API 权限。

为什么不是优先 Three-Legged

Three-Legged OAuth 通常需要创建 Schoology developer app，提供可访问的 App URL、登录或回调配置，并处理用户跳转授权。 这更适合嵌入 Schoology、安装到 Schoology App Center，或服务多个学生的正式应用。

单纯本地运行的 Mac 应用没有天然的公网回调服务，也不需要多用户授权体系，所以把 Three-Legged 作为后续产品化路径更合理。

凭据来源与验证路径

首选验证：个人 API key / Two-Legged OAuth

学生登录自己的 Schoology 账号后，先检查账号设置、API 或 Integration 相关入口是否能生成 consumer_key 和 consumer_secret。如果可以，就把密钥存入 macOS Keychain， 请求时令 oauth_token 为空字符串，并只读取学生本人有权限访问的数据。

产品化路径：developer app / Three-Legged OAuth

如果未来要做成正式 Schoology App，或给多个学生使用，则需要创建 Schoology developer app。 创建后可在 App Profile 的 Options -> API Info 获取 OAuth 1.0 credentials； 每个学生再通过授权流程产生自己的 oauth_token 和 oauth_token_secret。

兜底路径：浏览器插件或粘贴导入

如果学校没有开放学生个人 API key，且创建 developer app 的权限也不可用，可以考虑读取学生已经登录并可见的 Schoology 页面， 或保留粘贴/fixture 导入作为降级方案。

Two-Legged OAuth 管理员确认清单

目标说明：学生自己开发一个本地 macOS 桌面应用，只读取该学生本人在 Schoology 中已经有权限看到的信息， 例如课程、作业、截止日期、事件和成绩只读信息；不访问其他学生数据，也不修改 Schoology 数据。 这里要向管理员确认的是 Two-Legged OAuth / Personal API key 是否可用，不是申请 Schoology developer app 或 Three-Legged OAuth。

  1. 是否允许学生账号使用 Schoology API

    请确认学生角色是否可以启用 Access Schoology API 权限。如果不能给全部学生开放， 是否可以给一个测试学生账号或测试角色临时开放。

  2. 学生是否可以生成自己的 Personal API key

    请确认学生登录 Schoology 后，是否能生成该学生自己的 consumer_key 和 consumer_secret。这应该是用于 Two-Legged OAuth 的学生个人 key，不是学校管理员 key，也不是全校 integration key。

  3. API key 的获取入口

    请管理员提供本校 Schoology 中生成个人 API key 的具体路径。不同学区配置可能不同， 入口可能在 API、Integration 或账号相关页面。

  4. 确认这个 key 的权限边界

    请确认该学生 personal API key 是否只能访问该学生本人可见的数据；是否允许读取 users/me、学生加入的 courses / sections、section assignments、due dates / calendar events， 以及 grades 只读信息；是否禁止访问其他学生、老师或全校 roster 数据。

  5. 确认 Two-Legged OAuth 是否可用于学生 personal key

    应用会使用 OAuth 1.0a Two-Legged 方式：oauth_consumer_key 使用学生自己的 consumer key， oauth_token 为空字符串，每次请求重新生成 oauth_nonce 和 oauth_timestamp。请确认本校没有禁用这种 personal API key 使用方式。

  6. 确认 API Base URL / Schoology 域名

    请提供本校 Schoology 域名，并确认 API 请求应该使用的 base URL。如果本校有自定义域名、SSO 或特殊网关， 也请说明是否会影响 API 调用。

  7. 确认是否有测试账号或测试课程

    如果不方便直接用真实学生账号测试，请提供一个测试学生账号。测试账号最好加入一个测试课程， 并包含几个 assignments / due dates，方便验证 API 能否正常读取。

  8. 确认安全要求

    学生应用会把 consumer_secret 存在 macOS Keychain，不写进源码、不放进 SQLite、不上传到云端。 如果 key 泄露，请确认学生或管理员是否可以 revoke / regenerate key，并确认学校是否允许学生个人本地应用保存自己的 API credentials。

  9. 确认使用限制

    请确认是否有 API 调用频率限制，是否允许本地应用定时同步，例如每 15 分钟、30 分钟或手动同步； 也请确认是否禁止浏览器插件读取学生已经能看到的 Schoology 页面，作为 API 不可用时的兜底方案。

  10. 确认 2025 年后的 Personal API key 限制

    Schoology 更新说明提到 Personal API keys 不能访问其他用户数据。这个应用只访问学生本人数据， 请管理员确认这种用途是否被允许。如果 personal key 调用自己的课程或作业也返回 401， 需要判断是权限未开、key 类型不对，还是学校策略禁用了学生 API。

可直接发给管理员的简短说明

我想开发一个只给自己使用的本地 macOS 桌面应用，用来读取我自己在 Schoology 里已经能看到的课程、作业、截止日期和成绩只读信息。 计划采用 Schoology OAuth 1.0a Two-Legged OAuth / Personal API key 方式。请问学生角色是否可以启用 Access Schoology API 权限，并生成我自己的 personal consumer_key / consumer_secret？

应用只访问我自己的数据，不访问其他学生数据，也不会修改 Schoology 内容。请求中 oauth_token 会按 Two-Legged OAuth 要求传空字符串。 如果学生角色不能生成 personal API key，是否可以提供测试账号、临时测试权限，或说明学校是否禁止这种 API 使用方式？

Three-Legged OAuth 流程

适合“一个应用服务很多用户，每个用户只授权访问自己权限范围内的数据”。

应用识别当前 Schoology 用户。
检查是否已有该用户的 access token。
没有 token 时请求 /oauth/request_token。
跳转用户到 /oauth/authorize 批准授权。
回调后用 request token 换 access token。
用 access token 代表该用户调用 API。

Two-Legged OAuth 流程

适合“没有前台用户点击授权，而是后台系统或管理员集成访问 API”。

管理员或有权限角色获取 API consumer key。
服务端保存 consumer key 和 consumer secret。
每次请求生成新的 nonce 和 timestamp。
把 oauth_token 设置为空字符串。
生成 OAuth1 签名并发送 API 请求。
Schoology 根据该 key 所属身份判断权限。

典型应用场景

Three-Legged

老师教学工具

老师打开第三方教学插件，插件读取老师自己教授的班级、学生名单、作业或课程材料。 应用不能天然越过老师账号本身的权限边界。

Three-Legged

学生或家长应用

学生授权后查看自己的课程、作业和成绩；家长授权后查看与孩子相关的信息。 权限跟随当前授权用户，而不是应用想看什么就能看什么。

Two-Legged

学生个人 Mac 应用

学生自己开发并在本机运行的日程助手，只读取自己的课程、作业和截止日期。 如果学生账号能生成个人 API key，这条路径最直接。

Two-Legged

SIS 或教务系统同步

学校后台系统定时创建用户、更新学生资料、同步课程、处理注册关系或停用离校学生。 这类任务通常不适合让每个用户逐个授权。

Two-Legged

管理员报表和导出

学校管理员脚本或内部报表系统批量拉取全校数据。 是否能看到全校，取决于 key 所属账号是否具备 school-wide admin 等权限。

需要谨慎

跨学校第三方 SaaS

如果第三方应用要访问其他学校的用户数据，还会受到 Schoology App Center 审批和跨学校访问限制影响。 未批准应用可能收到 401。

权限问题

只查看某个学生

这通常不是 key 自身配置了“只读某学生”的 scope，而是请求所代表的用户本来只对该学生有访问权。

如何选择

应用是否面向很多 Schoology 用户，并且需要按当前用户权限访问数据？
Three-Legged
是否需要用户明确批准第三方应用访问自己的数据？
Three-Legged
是否是学生自己开发、自己在本机使用、只访问自己 Schoology 信息的 Mac 桌面应用？

Two-Legged

实现注意事项

Schoology 的权限判断不是单纯由 API key 上的细粒度 scope 决定，而是由 OAuth 身份、用户角色、学校或楼宇范围、课程或班级上下文，以及应用审批状态共同决定。

  • 遇到 403，通常表示当前身份没有权限执行该操作。
  • 遇到 401，可能是 token 过期、签名错误、nonce/timestamp 重复，或应用访问范围不被允许。
  • OAuth1 access token 最长有效期为 90 天，过期后需要重新授权。
  • Personal API keys 不能访问其他用户数据；跨学校访问需要关注 Schoology 的审批要求。
  • 学生本地应用不要把 consumer_secret 写进前端源码或公开仓库；在 macOS 上应使用 Keychain 保存。
  • 如果学生角色没有 Access Schoology API 权限，Two-Legged 路径会被学校策略阻断，需要切换到插件、粘贴或正式 developer app 路径。
